Docker 实现 DevOps

发表于 2020-12-20 | 分类于 Docker ， DevOps

1. 容器技术和 Docker 简介

1.1 什么是 Docker

传统的应用部署方式, 需要将应用单独部署在单独的物理机上, 存在如下问题:

  • 部署慢
  • 成本高
  • 资源浪费
  • 难以扩展迁移

在虚拟化技术出现后, 可以在原先的物理服务器上使用通过使用 Hypervisor 去完成物理资源的虚拟化(CPU, 内存, 磁盘, 网络等虚拟化), 在虚拟层之上在安装操作系统(虚拟机), 虚拟化技术拥有如下优点:

  • 资源池: 一个物理机的资源可以分配给多个虚拟机
  • 方便扩展: 物理机和虚拟机都可以动态扩展
  • 方便上云: 亚马逊 AWS, 阿里云等

但与此同时, 虚拟化技术依然存在着局限性:

  • 每个虚拟机都是完整的操作系统, 要给其分配资源, 当虚拟机同一台物理机上的虚拟机数量过多时, 操作系统本身消耗的资源就会过多.

容器是在软件层面的隔离, 而虚拟化是在物理资源层面的隔离

容器是对软件和其依赖的标准化打包, 能够在共享同一个 Kernel 的情况下完成应用隔离, 可以运行在所有主流操作系统之上.

1.2 Docker 的作用

  • 简化配置: 容器最重要的功能, 能够实现将源代码, 运行环境和配置一起打包到容器中, 容器可以运行在不同的环境中.
  • 代码流水线管理
  • 提高开发效率: 除了简化配置, 还可以最大限度保持本地环境, 测试环境和开发环境一致
  • 隔离应用
  • 整合服务器
  • 调试能力
  • 多租户
  • 快速部署

容器解决了什么问题:

  • 开发人员和运维人员之间的沟通问题: 容器可以将应用打包, 可以像集装箱一个去打包, 运输应用, 并部署在任意环境.
  • 在开发和运维之间搭建了一个桥梁, 是实现 DevOps 的最佳解决方案.

1.3 Docker 和 Kubernetes 的关系

Kubernetes 是一个容器编排工具, 用于管理容器的生命周期, 包括容器的创建, 管理, 调度, 运维等

3. Docker 的镜像和容器

3.1 概览

Docker 提供了一个开发, 打包, 运行应用的平台, 将应用和底层基础设施隔离开来.

Docker 包括如下几部分:

  • 后台进程(dockerd)
  • REST API Server
  • Cli 接口

底层技术支持:

  • Namespace: 隔离 pid, net, ipc, mnt, uts
  • Control Groups: 做资源限制
  • Union file systems: Container 和 image 的分层

3.2 Image

BaseImage: 在 Linux Kernel 之上为每个 Linux 发行版制作的基础镜像

  • 文件和 meta data 的集合(root filesystem)
  • 是分层的, 并且每一层都可以改变和删除文件, 成为一个新的 image
  • 不同的 image 可以共享相同的 layer
  • image 本身的只读的

image 的获取方式:

  • build from Dockerfile
  • pull from Registry

3.6 Dockerfile 语法

FROM: 指定 base image

应尽量使用官方 image

  • LABEL: 定义 image 的 mata data

RUN: 容器启动时指定的命令

为了美观, 负责的 RUN 请用反斜杠换行
每个 RUN 会单独生成一个 layer, 避免无用分层, 合并多条命令成一行

WORKDIR: 设定当前工作路径

尽量使用 WORKDIR 而不是 RUN cd
尽量使用绝对目录

ADD and COPY: 将本地文件移动到镜像的指定路径

大部分情况下, COPY 优于 ADD, ADD 除了 COPY 还有额外功能(解压)
添加远程文件/目录请使用 curl 或者 wget

ENV: 设置环境变量

尽可能使用 ENV 增加文件的可维护性

RUN, CMD 和 ENTRYPOINT

  • RUN: 执行命令并创建新的 Image Layer
  • CMD: 设置容器启动后默认执行的命令和参数
  • ENTRYPOINT: 设置容器启动时运行的命令

用上述两个 Dockerfile 分别执行 build 和 run 启动容器后, shell 格式会打印出 hello docker, 而 exec 格式会打印出 hello $name. shell 模式是默认在 shell 中执行的, 而 exec 只是单纯执行命令, 如果想让 exec 在 shell 中执行, 上述Dockerfile 应改为:

  • CMD:
    1. 容器启动时默认执行
    2. 如果 Docker RUN 指定了其他命令, CMD 命令被忽略
    3. 如果定义了多个 CMD, 只有最后一个会执行
  • ENTRYPINT:
    1. 让容器以应用程序或服务的形式运行
    2. 不会被忽略, 一定会执行
    3. 最佳实践: 写一个 shell 脚本作为 ENTRYPOINT

4. Docker 的网络

4.1 Docker 容器网络

4.1.1 网络相关背景知识

  • 协议: 协议就是一种语言, 支持同一协议的双方就能够理解对方传递的信息.
  • 网络接口: 拥有一个地址, 代表了一个位置, 接口可以类比成真实世界中带有地址的位置, 网络接口像一个信箱.
    一个网路接口拥有一个 IP 地址, IP 地址是由 IP 协议定义的, 一般来说计算机拥有两种网络接口:
    1. 以太网接口: 用来连接其他接口和进程
    2. 本地回环接口: 利用网络协议来与同一台机器上其他进程童新师非常有用.
  • 端口: 实际上端口仅仅是一个数字, 是 TCP 协议的一部分, 记录在 TCP 上的端口信息相当于写在信封上的名字, 能够用来区分收件人.
  • 网络: 每个网络接口代表了网络中的一个节点, 接口连接在一起就定义了网络, 并且这个连接决定了接口的 IP 地址.

  • 端口转发:

4.1.2 本地 Docker 网络的拓扑结构

Docker 使用操作系统的底层特性构建了一个特殊的, 可定制的虚拟网络拓扑结构.

每个容器各自拥有一个本地回环接口和一个分离的以太网接口, 其中以太网接口连接着主机 namespace 上的另一个虚拟接口, 这两个互连的接口在宿主机网络栈和每个容器的网络栈之间建立了连接, 每个容器都被赋予了一个唯一的私有 IP 地址, 从外部不能直连. 网络连接需要经过 Docker 网桥接口路由到另一个网络, 这个网桥接口成为 docker0(类似家庭网络中的路由器), 为每个容器创建的虚拟接口都会连接到 docker0, 这样他们就构成了一个网络, 这个网桥接口 docker0 会连接到主机所连接的网络上.

4.1.3 四种网络容器原型

所有的 Docker 容器都要符合这四种原型中的一种, 这些原型定义了一个容器如何与其他的本地容器, 主机网络进行通信, 每种原型有不同的目的, 可以认为它们拥有不同程度的隔离, 使用 Docker 容器时, 要根据实际场景靠谱使用哪个原型, 四种容器按排列顺序隔离程度递减:

Closed 容器

不允许任何的网络流量, 容器中的进程只能够访问回环接口, 合适进程只需要和本地进程通信的场景. 可以在 docker run 后添加 --net none 作为参数告诉 Docker 创建一个 Closed 容器.

Bridged 容器

bridged 容器开放了网络的隔离程度, 可定制型最高, 被认为是最佳实践, bridged 拥有两个接口:

  • 本地回环接口: 用于本地通信
  • 私有接口: 通过网桥连接到主机的其他容器
Joined 容器
Open 容器

4.2 NetworkNamespace 原理

在创建容器的时候, Docker 首先会为容器创建一套目标容器独占的 Network Namespace, 以此实现 Linux 容器的网络隔离

ip a 的输出就是一个 network namespace

同理, 创建多个容器时, 每个容器都会有一个单独的 Network Namespace

尽管不同容器之间 Network Namespace 是相互独立的, 但可以通信.

现在有两个 NetworkNamespace: test1 和 test2, 各自只有一个 lo 回环口, 现在通过创建一对 Veth pair 将两个 Namespace 连起来

4.3 Docker Bridge0

  1. 两个容器之间是如何完成通信的?
  2. 容器是如何访问 Internet?

对于容器 test1, 是连接到 bridge 网络上的, 此时的网络拓扑是, 每个 docker 容器各通过一个 veth 连接到 docker0 这个 bridge 上, 原理类似于路由器. 每个 docker 容器只需要通过 docker0 就可以相互通信, 每个容器分别是不同的 NetworkNamespace, docker0 则位于宿主机的默认 NetworkNamespace 中

默认情况下, Linux 主机通过 eth0 可以访问外网, 容器的数据包经过 docker0 时, 如果可以完成 NAT 转换, 就可以转换成 eth0 的地址, 此时可以作为 Linux 主机的数据包去访问外网, NAT 转换使用 IPtable 实现的.

4.4 Docker容器之间的 link

假设有一个后台系统, 后端应用和数据库都部署在 Docker 容器中, 传统方式下我们需要先启动数据库容器, 启动后查看其 ip 地址并配置到后端应用容器中. 但这种方式无法完成应用自动部署

Docker 中可以给容器起一个名字, 通过名字代替 ip 来访问指定容器. 这就是 Docker 的 link 机制, 在创建后续容器时, 可以 link 到前面的容器上, 这样在访问原有容器是可以通过 name 代替 ip 地址.

需要注意的是 link 参数是单向的, 此时只有 test2 link 到了 test1, 反之并不成立. 假设 test1 中有端口为 3306 的 MySQL 服务, 此时在 test2 中也可以通过 mysql -h test1 -P 3006 -u root -p 进行访问.
